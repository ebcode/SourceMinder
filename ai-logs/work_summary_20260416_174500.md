## Changes Made

### 1. `scalar @arr` coercion тАФ verified, no change needed

The plan flagged a risk that `scalar @arr` (unary context operator) might
collide with `perl_symbols.scalar` (the `$` sigil node). Created
`tmp/test_scalar_coerce.pl` and ran with `--debug`.

**Finding:** `perl_symbols.scalar` is registered via
`ts_language_symbol_for_name(language, "scalar", 6, true)` with `named=true`.
The `scalar` keyword in `scalar @arr` is an anonymous token, so it has a
different `TSSymbol` from the named sigil node. No collision. The debug output
confirmed `visit_node` reports "no handler for 'scalar'" on the keyword тАФ the
`@arr` operand is the only thing indexed from the expression.

### 2. `@EXPORT_OK` bare list form

**Problem:** `our @EXPORT_OK = ('foo', 'bar', 'baz')` (string-literal list)
was not producing `EXP` entries. Only the `qw(...)` form was handled.

**AST structure discovered (via `--debug`):**
- Multiple items: `list_expression` with `string_literal` children
- Single item: bare `string_literal` (no `list_expression` wrapper)

**Fix in `perl/perl_language.c`:**
- Added `list_expression` to the `perl_symbols` struct and `init_perl_symbols`
  (length 15)
- Extended the EXPORT detection block in `handle_variable_declaration` to
  scan for `list_expression` / bare `string_literal` siblings after the
  existing `quoted_word_list` loop. Each nested `string_content` becomes an
  EXP entry with `definition="1"`.

**Verification:** `tmp/test_export.pl` with both forms тЖТ `foo`, `bar`, `baz`,
`qux` all indexed as EXP.

### 3. Anonymous sub тЖТ `FUNC`

**Decision:** `my $greet = sub { ... }` now indexes `$greet` as `CONTEXT_FUNCTION`
(with `modifier="my"` and `type=PERL_TYPE_SCALAR`), not `VAR`. A FUNC is what
the user searches for; the `$` sigil is an implementation detail visible at
call sites. Kept both `modifier` and `type` columns populated so no information
is lost.

**Fix in `handle_variable_declaration`:** in the final "walk all children"
section, when the sigil is a `scalar` and the `assignment_expression` parent
has an `anonymous_subroutine_expression` sibling, emit a `FUNC` entry
directly and skip the normal `index_sigil_node` call.

**Verification on `closures.pl`:** `$size = sub {...}`, `$slow_double = sub {...}`,
`$in_range = sub {...}` all now show as FUNC with `my | scalar`. RHS function
calls (`$fast_double = memoize(...)`, `$evens = counter_from(...)`) correctly
stay as VAR тАФ no false positives.

### 4. `qi` directory filter bug

**Problem:** `-f tools/sources/perl/` (full relative path with trailing slash)
returned 0 matches, while `-f sources/perl/` worked.

**Root cause in `query-index.c::process_file_pattern`:** `needs_prefix=1` path
emitted SQL `%/tools/sources/perl/`. The stored directory is
`tools/sources/perl/` (no leading `/`), so the pattern required a character
before `tools` that doesn't exist.

**Fix:** Keep the `%/` boundary prefix for single-component names
(`perl/` тЖТ `%/perl/`, prevents matching `myperl/`), but drop to plain `%`
prefix when `dir_part` already contains a `/`
(`tools/sources/perl/` тЖТ `%tools/sources/perl/`).

  in that exact directory" row alongside the partial-path `shared/` example
- `README.md` File Filtering sections (two of them): added the full-path
  form to both example lists

### 7. Complex dereference bug: `{$arr}` symbol name

**Problem:** `@{$arr}[0..$n - 1]` in `data-structures.pl` line 111 was
producing a symbol named `{$arr}` (with braces) instead of `arr`.

**Investigation via `--debug` + temporary debug prints in `index_sigil_node`:**
the tree-sitter-perl AST for `@{$arr}` is:

```
array
  @                  (anonymous)
  varname("{$arr}")  (named, text spans the whole block)
    block            (named, child of varname)
      scalar
        $
        varname("arr")
```

The prior complex-dereference fix walked the `varname`'s direct children
looking for `scalar`/`array`/`hash` тАФ but the sigil is nested inside a `block`,
so no direct child matched and extraction silently failed, falling back to
raw text.

**Fix:** Added a `find_inner_sigil` helper that recursively searches descendants
for the first `scalar`/`array`/`hash`/`container_variable` node. Both
`index_sigil_node` and `extract_sigil_varname` now use it in their
complex-dereference branches.

**Verification:** `tmp/test_slice.pl` тАФ all four forms work:
- `@{$arr}[0..2]` тЖТ `arr` VAR array def=0 (via slice_container_variable)
- `@{$arr}` standalone тЖТ `arr` VAR scalar def=0 (via outer array sigil)
- `${$arr}[0]` тЖТ `arr` VAR array def=0 (via array_element_expression)
- `data-structures.pl:111` тЖТ `arr` VAR array, no `{$arr}` bug
