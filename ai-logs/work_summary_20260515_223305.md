## Issues Encountered

### File filter `-f` silently matched nothing due to path prefix mismatch

- **Symptom:** `-f alacritty_terminal/` returned "No files matched" even though those files were indexed
- **Root Cause:** Indexed paths include the full prefix from the repo root (e.g. `./alacritty/alacritty_terminal/...`). The filter does substring matching against the full path, so a partial directory name without the leading prefix doesn't match
- **Fix:** Run `qi % -i file | head -5` first to see real path structure, then construct the `-f` pattern from actual paths
- **Regression Risk:** LOW — this is a UX friction point, not a bug; the workaround is reliable
- **Follow-Up:** qi could show sample matching paths in the "no files matched" error message to make the issue self-diagnosing

### Struct field (prop) queries returned no results

- **Symptom:** `qi % -i prop -p Term --def` returned nothing for alacritty's `Term` struct despite its fields being visible in the `-e` expansion
- **Root Cause:** Uncertain — either the Rust indexer isn't emitting PROP entries for struct fields in all cases, or the parent-column matching for `Term` didn't align with how the impl is stored
- **Fix:** Fell back to `-e` on the class definition, which showed fields inline
- **Regression Risk:** LOW
- **Follow-Up:** Worth investigating whether PROP entries are being emitted for Rust struct fields consistently

## Notes / Design Decisions

- **`-i file` as hypothesis generator:** In a well-named codebase, the flat list of indexed filenames is often the densest possible summary of what the system does. Running it first and reading the module names before any other query is the right starting move for an unfamiliar project — not just an orientation step
- **Multi-symbol `-e` is the highest-leverage single query pattern:** Five interrelated struct definitions in one call gives you a type hierarchy you can read as a unit. The alternative requires multiple Read calls and mental reassembly
- **`--raw` belongs in every project's qi guide:** It solves a real problem (the annotated output is uneditable) with a clean design (two modes, one flag). Worth surfacing explicitly in project documentation alongside `-e`
- **qi vs ctags:** Not competitive for exploration — ctags gives definition locations, qi gives codebase shape. The comparison with grep is more interesting: grep wins for literal text, qi wins for semantic structure. Neither fully replaces the other

## Tooling Ideas

1. **`--callers` / `--callees` flags:** The two-query pattern (`qi fn -i call` for callers, `qi % -i call --within fn` for callees) is common enough to deserve a dedicated shorthand. A single `qi create_initial_window --callers` returning call sites with one line of context each would compress the most frequent call-graph navigation pattern into one command
2. **Path-aware "no files matched" error:** When `-f` returns no matches, show two or three sample indexed paths so the user can immediately see what prefix format is needed. Would have eliminated the main friction point in both exploration sessions without any changes to core functionality

## Documentation Ideas

1. **`--raw` prominently in the main `qi --help` output:** The flag is listed but its specific use case (edit anchoring) isn't explained. A one-line note — "use with `-e` to produce source lines suitable as an edit `old_string`" — in the help text would make it discoverable without needing a project guide to surface it
2. **A "first five queries for a new project" quick-start in the Rust guide:** The optimal cold-start sequence (`-i file` → `--toc` on entry points → multi-symbol `-e` on key types → `-i trait --def` → `--within` on the main function) wasn't obvious from the guide as written. A short "if you know nothing, run these five things in order" section at the top of `docs/RUST_GUIDE.md` would serve both new users and returning users starting on an unfamiliar codebase
