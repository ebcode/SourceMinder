## Issues Encountered

### Attribute extraction returned empty clue column

- **Symptom:** `#[derive]`, `#[inline]`, `#[test]` etc. were never populated in the clue column even though sibling `attribute_item` nodes existed in the AST
- **Root Cause:** Used `ts_node_child_by_field_name(sib, "attribute", ...)` but in tree-sitter-rust the inner `attribute` is a regular child node, not a named field
- **Fix:** Iterate `attribute_item`'s children looking for type `"attribute"` directly; then iterate that node's children for the first `identifier`/`scoped_identifier` (the attribute path)
- **Regression Risk:** LOW
- **Follow-Up:** NONE

### Bodyless trait methods and FFI declarations not indexed

- **Symptom:** Trait method signatures like `fn read(&self) -> ...;` (no body) and FFI declarations inside `extern "C" { ... }` produced zero index entries
- **Root Cause:** These items use a distinct AST node type, `function_signature_item`, which had no dispatch handler
- **Fix:** Added `handle_function_signature_item` (mirrors `handle_function_item` minus body processing) and wired it into `visit_node`
- **Regression Risk:** NONE
- **Follow-Up:** NONE

### Pattern variable bindings missed (match arms, if-let, for-loops)

- **Symptom:** `match Some(Ok(v))`, `if let Some(value) = opt`, `for (name, count) in pairs` produced no VAR entries for the bindings
- **Root Cause:** No handlers for `match_expression`, `for_expression`, or `let_condition`; the let-only pattern walker also didn't traverse `tuple_struct_pattern` / `struct_pattern` / `or_pattern` / etc.
- **Fix:** Generalized the pattern walker into `index_pattern_identifiers` that handles all pattern node types, skips constructor identifiers (uppercase-start) and `scoped_identifier` (always a constant/variant reference, never a binding), and dropped the constructor child of `tuple_struct_pattern`/`struct_pattern`. Added handlers for `for_expression`, `match_expression`, and `let_condition`
- **Regression Risk:** LOW — uppercase-skip heuristic is a Rust convention; if a user violates it, their lowercased-binding would still be captured but uppercase-named bindings (rare, frowned upon) would be skipped
- **Follow-Up:** NONE

### Impl target recorded with generic parameters in symbol

- **Symptom:** `impl<'a, T> Wrapper<'a, T> { ... }` produced a CLASS entry with symbol `Wrapper<'a, T>` instead of `Wrapper`, breaking `qi Wrapper -i class`
- **Root Cause:** Used `safe_extract_node_text` directly on the `type` field, which returns the full type expression
- **Fix:** Added `extract_base_type_name` that peels off `generic_type` wrappers and `scoped_type_identifier` qualifiers to find the underlying `type_identifier`. Used for both the impl target and the trait name
- **Regression Risk:** NONE
- **Follow-Up:** NONE

### `pub` originally placed in `modifier` column

- **Symptom:** Composability awkward — `qi -m unsafe` matched only `unsafe` while `qi -m '%async%'` was needed because `modifier` was `"pub async"`. No way to say "all public functions"
- **Root Cause:** First pass mashed visibility and behavior keywords into one column for simplicity
- **Fix:** Split `extract_fn_modifiers` to drop visibility; visibility now flows via `extract_visibility` to the `scope` column. All item handlers updated. Added `RUST` to the `scope`/`namespace` enablement gate in `shared/column_schema.def`
- **Regression Risk:** NONE (Rust-only change; other languages unaffected)
- **Follow-Up:** NONE

## Residual Risks / Not Verified

- Not yet tested against a real-world Cargo project (only synthetic test files). Likely surfaces edge cases around `proc_macro`, `async-trait`, complex `where` clauses, and patched/aliased identifiers
- The README hasn't been updated to advertise Rust in the "Languages currently implemented" list or the grammar clone list
- The Rust grammar version isn't yet recorded in `rust/grammar_version.h` for the no-Rust-enabled build path (only generated when Rust is enabled at configure-time, which is the correct behavior)
- Block comments report a single line number (the start line) for every word — adequate for indexing but slightly less precise than per-line tracking
- The "skip uppercase-start identifier in pattern position" heuristic for filtering constructors will mis-skip a binding named like `MyVar` (against convention). Tradeoff favors precision over completeness

## Notes / Design Decisions

- **`scope` vs `modifier` split:** `pub`/`pub(crate)`/`pub(super)` → `scope` (matches PHP/TS visibility semantics). `async`/`unsafe`/`const`/`extern` → `modifier`. `await` (on calls) → `modifier`. This was a mid-session refactor after `column_schema.def` review made the cleaner semantics clear
- **Impl target tracking:** Used a file-static `g_current_impl` buffer that's saved/restored around `process_children` of an impl/trait/enum body. Cleaner than threading it through every function signature
- **Constructor heuristic in patterns:** Skip any pattern-position identifier starting with uppercase, plus all `scoped_identifier`. Rust's naming convention makes this reliable in practice
- **AST exploration cadence:** Built the AST explorer first, then iteratively wrote handlers against the real tree-sitter output for each construct. Avoided several wrong guesses about field names (e.g., `attribute_item` exposes `attribute` as a child, not a field)
- **Macros indexed as `CONTEXT_FUNCTION`** (with `clue="macro_rules!"`) since they're callable; macro invocations as `CONTEXT_CALL` with `clue="macro!"`. Consistent with Python's approach for decorators-as-calls
